# Compliance: FIPS 140-3

FIPS 140-3 (Federal Information Processing Standard 140-3) defines security requirements for cryptographic modules used by US federal agencies and their contractors. It is also required by frameworks that reference it (FedRAMP, DoD IL, HIPAA for federal contractors, some financial regulations).

FIPS compliance has two aspects:
1. **Algorithm compliance** — only approved algorithms and key sizes used
2. **Module validation** — the cryptographic library is FIPS 140-3 validated (CMVP certificate)

Add `fips-140-3` to `stacks:` for any project serving US federal agencies, FedRAMP authorized systems, or contracts requiring FIPS compliance.

---

## Scope

Identify all cryptography in this feature:
- Data encryption (at rest, in transit)
- Hashing (data integrity, password storage, digital signatures)
- Key generation and key derivation
- Random number generation
- Digital signatures / certificates
- TLS configuration

---

## Approved Algorithms (FIPS 140-3 / SP 800-131A Rev 2)

### Symmetric Encryption
- [ ] AES-128, AES-192, or AES-256 used — no DES, 3DES (Triple-DES deprecated 2024), RC4, Blowfish, or any other symmetric cipher

### Asymmetric Encryption / Key Exchange
- [ ] RSA ≥ 2048-bit, or ECC (P-256, P-384, P-521) used
- [ ] No RSA < 2048-bit, no DSA < 2048-bit after 2015
- [ ] No Diffie-Hellman with group size < 2048 bits

### Hashing
- [ ] SHA-256, SHA-384, or SHA-512 used for integrity and digital signatures
- [ ] SHA-3 family (SHA3-256, SHA3-512) also approved
- [ ] **Prohibited**: MD5 (no use, including non-security contexts), SHA-1 (prohibited for digital signatures; allowed only for legacy verification of existing signed data — flag any new use)

### Password Hashing / Key Derivation
- [ ] PBKDF2 (with SHA-256 or SHA-512, ≥ 10,000 iterations), bcrypt, or scrypt used
- [ ] Argon2 accepted where FIPS mode allows SP 800-63B guidance
- [ ] No MD5 or unsalted SHA-1 for password storage
- [ ] KDF: HKDF (SP 800-56C) or SP 800-108 KDF used for key derivation — no custom KDFs

### Random Number Generation
- [ ] CSPRNG used for all security-sensitive randomness (keys, tokens, nonces, IVs)
- [ ] Source: OS CSPRNG (`/dev/urandom`, `crypto.rand`, `os.urandom`) or FIPS-validated DRBG
- [ ] No `math/rand` (Go), `random.random()` (Python), or `Math.random()` (JS) for security contexts

### TLS Configuration
- [ ] TLS 1.2 or TLS 1.3 only — no TLS 1.0, TLS 1.1, SSL 2.0/3.0
- [ ] TLS 1.3 preferred (always uses forward secrecy)
- [ ] Cipher suites restricted to FIPS-approved: AES-GCM, AES-CBC with HMAC-SHA2; no RC4, no 3DES, no export ciphers
- [ ] Certificate uses RSA ≥ 2048 or ECDSA P-256/P-384; SHA-256 or stronger signature

### Digital Signatures
- [ ] RSA with PKCS#1 v2.1 (PSS padding) or ECDSA (P-256, P-384, P-521)
- [ ] Signature uses SHA-256 or stronger hash
- [ ] No MD5withRSA, no SHA1withRSA for new signatures

## Validated Cryptographic Modules (CMVP)

If strict FIPS 140-3 validated module usage is required (federal contracts, FedRAMP High):

- [ ] Cryptographic operations use a CMVP-validated module (check [NIST CMVP](https://csrc.nist.gov/projects/cryptographic-module-validation-program))
  - OpenSSL with FIPS provider (validated)
  - AWS KMS (validated)
  - Azure Key Vault HSM (validated)
  - Google Cloud HSM (validated)
  - BoringCrypto / BoringSSL (Go `GOEXPERIMENT=boringcrypto`)
- [ ] Cryptographic keys generated within or imported into a validated module — no keys generated in userland then loaded into the module
- [ ] Module operates in FIPS-approved mode (some libraries have a FIPS mode flag that must be explicitly enabled)

---

## Key Management

- [ ] Encryption keys never stored alongside the data they protect
- [ ] Keys stored in a validated HSM or KMS (not in application config or source code)
- [ ] Key rotation policy defined: max key age, rotation trigger (compromise, personnel change)
- [ ] Key destruction procedure documented (crypto-shredding for storage deletion)
- [ ] Key access logged: who used which key, when

---

## Common Violations (flag as CRITICAL)

- MD5 used anywhere — even for non-security purposes in a FIPS context
- SHA-1 used for digital signatures or new password hashing
- DES or 3DES in any cryptographic path
- `math/rand`, `random.random()`, or `Math.random()` used to generate tokens, IDs, or keys
- RSA < 2048 bits
- TLS 1.0 or 1.1 enabled in server config
- RC4 or NULL cipher suite enabled in TLS config
- Custom or homebuilt cryptographic implementation (use library functions only)
- Encryption key stored in source code, config file, or environment variable (use KMS/Vault)
