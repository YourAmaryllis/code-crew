# Stack: Terraform (Generic)

These conventions apply to all Terraform projects regardless of cloud provider.
For AWS-specific patterns (ECS, Secrets Manager, OIDC, S3 backend), see `terraform-aws`.

## Toolchain

| Item | Tool/Version |
|------|-------------|
| IaC tool | Terraform 1.7+ |
| Lint | `terraform validate`, `tflint` |
| Format | `terraform fmt` |
| Security scan | `tfsec` / `checkov` |
| Plan review | manual — plan output reviewed by architect before apply |
| Apply | manual only — never automated in CI for staging/prod |
| Pre-commit | `terraform fmt`, `tflint` |

## Directory Layout

```
infra/
  modules/
    <module-name>/
      main.tf          # resources
      variables.tf     # input variables (all documented)
      outputs.tf       # outputs (all documented)
      versions.tf      # required_providers + terraform version constraint
      README.md        # generated by terraform-docs
```

## Naming Conventions

All resources include environment prefix and follow kebab-case:

```hcl
resource "some_service" "api" {
  name = "${var.env}-${var.service_name}-api"
}
```

| Convention | Pattern |
|-----------|---------|
| Resource name | `${var.env}-${var.service_name}-<type>` |
| Parameter store keys | `/<project>/${var.env}/<service>/<key>` |
| Storage buckets | `${var.env}-${var.project}-<purpose>` |
| IAM / RBAC roles | `${var.env}-${var.service_name}-<role>` |

For provider-specific naming (e.g. SSM paths, IAM role naming on AWS), see the provider stack doc.

## Variable Conventions

All variables documented with `description`. Required inputs have no default — omitting them is an error.

```hcl
variable "env" {
  type        = string
  description = "Deployment environment (dev, staging, prod)"
  validation {
    condition     = contains(["dev", "staging", "prod"], var.env)
    error_message = "env must be dev, staging, or prod"
  }
}
```

Standard inputs every module must accept:

| Variable | Type | Purpose |
|----------|------|---------|
| `env` | string | Environment (dev / staging / prod) |
| `service_name` | string | Service identifier |
| `project` | string | Project name for namespacing |

No defaults for required inputs — forces explicit per-environment configuration.

## Apply Governance

- CI runs `terraform plan` on every PR that touches `infra/` — plan output is a PR artifact
- `terraform apply` is **never automated** — a human runs it after reviewing the plan
- Apply order follows dependency graph: shared infra → service infra (document this in the project's `infra/README.md`)
- Drift checks run on a schedule (`terraform plan` in staging and prod) to verify no manual changes

## Security Scanning

`tfsec` and `checkov` run on every Terraform PR:

```bash
tfsec infra/
checkov -d infra/ --framework terraform
```

Common findings to fix before merge:
- Open ingress rules (allow all inbound on any port)
- Unencrypted storage (buckets, volumes, databases)
- Missing `ReadOnlyRootFilesystem` on container definitions
- Missing encryption at rest for secrets/config stores
- Public IP assignment on compute resources that don't need it

## Module Encapsulation Philosophy

Modules hide provider complexity — project code deals in high-level intent, not provider API details.

Every module must provide:
- **Sensible defaults** with override capability
- **Consistent variable interface** across similar module types
- **Automatic wiring** of related resources (firewall rules, IAM/RBAC, encryption, etc.)

Project-level `main.tf` should read like a list of capabilities, not a list of provider resources.

## Access Policy and Deploy Policy Pattern

Every module that creates managed resources exposes two policy outputs:

| Output | Purpose | Typical permissions |
|--------|---------|-------------------|
| `access_policy` | Other resources **use** what this module created | Read, write, get, list |
| `deploy_policy` | CI/CD pipeline **manages** what this module created | Create, update, delete, tag |

```hcl
# outputs.tf — every module follows this pattern
output "access_policy" {
  description = "Policy for other modules to access resources created here."
  value       = <provider_policy_resource>.access.arn_or_id
}

output "deploy_policy" {
  description = "Policy for CI/CD pipelines to manage resources created here."
  value       = <provider_policy_resource>.deploy.arn_or_id
}
```

Usage at project layer: attach `access_policy` to services that consume the resource; attach `deploy_policy` to the CI/CD identity. See `terraform-aws` for the AWS IAM implementation.

## Infrastructure Layering

```
┌──────────────────────────────────┐
│  Project layer (infra/envs/*)    │  ← project-specific config only
├──────────────────────────────────┤
│  Opinionated modules             │  ← pre-packaged standard deployments
├──────────────────────────────────┤
│  Base modules (infra/modules/*)  │  ← reusable, generic
├──────────────────────────────────┤
│  Provider resources              │  ← Terraform providers
└──────────────────────────────────┘
```

| Layer | Responsibility | Change frequency |
|-------|---------------|-----------------|
| Project | Values and customisations per env | Frequent |
| Opinionated | Standard deployment bundles | Occasional |
| Base modules | Generic, reusable infrastructure | Rare |
| Provider resources | Raw Terraform | Very rare |

## No Direct Provider Resources at Project Layer

Project-level code (`infra/environments/*/`) must **never** declare raw `resource` blocks for managed infrastructure. All infrastructure goes through modules.

**❌ Anti-pattern:**
```hcl
# infra/environments/prod/main.tf — DON'T DO THIS
resource "some_database" "main" {
  name = "prod-db"
  # missing: encryption, backup, tagging, firewall rules
}
```

**✅ Correct:**
```hcl
module "database" {
  source   = "../../modules/database"
  env      = var.env
  name     = "main"
  # encryption, backup, tagging handled by the module
}
```

Acceptable exceptions at the project layer:
- Policy attachment resources — connecting module `access_policy`/`deploy_policy` outputs to identities
- `data` sources — referencing pre-existing resources
- `output` blocks

If you need functionality not covered by an existing module, extend the module — never add raw resources to project code.

## Required Resource Tags

All resources must carry a standard tag set for cost visibility, ownership, and compliance:

```hcl
tags = {
  env        = var.env
  service    = var.service_name
  managed-by = "terraform"
  project    = var.project
  cost-center = var.cost_center   # billing identifier / WBS code
}
```

Add `cost_center` as a required variable in every module that creates billable resources. Untagged resources are flagged by `tfsec`/`checkov`.

## Security and Data Classification

Storage modules (databases, object stores, caches, queues) require two classification inputs that drive automated security controls:

```hcl
variable "confidentiality_level" {
  type        = string
  description = "Data confidentiality: c1 (public) → c4 (strictly confidential)"
  validation {
    condition     = contains(["c1", "c2", "c3", "c4"], var.confidentiality_level)
    error_message = "Must be c1, c2, c3, or c4"
  }
}

variable "privacy_level" {
  type        = string
  description = "Privacy classification"
  validation {
    condition     = contains(["non_personal", "personal", "sensitive_personal"], var.privacy_level)
    error_message = "Must be non_personal, personal, or sensitive_personal"
  }
}
```

Controls applied by classification level:

| Control | C1/C2 + non-personal | C3/C4 or personal | sensitive_personal |
|---------|---------------------|-------------------|-------------------|
| Encryption at rest | — | ✅ | ✅ |
| Encryption key rotation | — | ✅ | ✅ |
| Audit logging | — | ✅ | ✅ |
| Data retention policy | — | ✅ | ✅ |
| Data masking | — | — | ✅ |
| Region restriction | — | ✅ (if personal) | ✅ |

Modules compute required controls internally; callers only set the two classification variables.

## Opinionated Modules

For standard deployment patterns, provide opinionated modules that bundle base modules with sensible defaults:

- **Use opinionated modules** for common patterns (e.g. compute service + registry + config store + load balancer)
- **Use base modules** when fine-grained control or non-standard configuration is needed

An opinionated module reduces hundreds of lines at the project layer to tens. Prefer opinionated modules for anything deployed more than once.

```hcl
# opinionated: one call deploys a full service stack
module "api_service" {
  source       = "../../modules/service-opinionated"
  env          = var.env
  service_name = "api"
  image_tag    = var.image_tag
  # firewall rules, RBAC, load balancer target, log shipping
  # all handled by the opinionated module
}
```
